The task
Build the backend for a reimbursements management tool for one of our partner organizations.
The whole point of the exercise is role-based access control: who can see what, and who can do what.
Stakeholders & roles
Stakeholder Role string Notes
Employee EMP Default role for every new account.
Reporting Manager RM A group of employees reports to one RM.
Accounts Payable Executive APE Handles payable-side approvals.
Chief Financial Officer CFO Root user. Seeded, not registered.
CFO (ROOT) CREDENTIALS — SEED EXACTLY THIS
email cfo@org.com
password CFO#ORG@April2026
Organization structure
A group of employees reports to exactly one RM.
Every employee reports to exactly one RM.
There is no direct relationship between:
EMP and APE
RM and APE
APE and CFO

API specification
All endpoints are under the /rest prefix. Authentication is cookie-based: the login response sets the
auth cookie, and protected endpoints read it automatically. Follow these routes, request bodies, and
response shapes precisely.
PUBLIC ENDPOINTS
POST /rest/onboardings/register
Self-service registration. Every account created this way starts with the default EMP role.
REQUEST BODY (JSON)
Employee's name , email , and password .
RULES
Only allow emails on the org.com domain. Assume the email is valid and belongs to the user —
any SMTP / verification work is out of scope for this task.
POST /rest/onboardings/login
REQUEST BODY (JSON)
Employee's email and password .
RULES
Only allow emails on the org.com domain.
POST /rest/onboardings/logout
No explicit payload — relies on the auto-tagged cookies sent with the request.
PROTECTED ENDPOINTS
POST /rest/roles/assign
Assigns a role to a user. We currently have a fixed set of roles: EMP , RM , APE , CFO .
WHO CAN CALL THIS
Only the CFO.
REQUEST BODY (JSON)
userId , role (the exact role string).

Assigns an employee to a reporting manager.

Only the CFO.

DELETE /rest/employees/assign
Removes the assignment between an employee and a reporting manager.

Only the CFO.

REIMBURSEMENT STATUS
A reimbursement's status is one of: PENDING , REJECTED , APPROVED .
POST /rest/reimbursements
Raises a new reimbursement.
WHO CAN CALL THIS
Only an EMP can create a new reimbursement.
REQUEST BODY (JSON)
title , description , amount .
BEHAVIOR
A newly raised reimbursement has status PENDING .
PATCH /rest/reimbursements
RULES BY ROLE
EMP cannot edit a reimbursement once it has been raised.
RM can approve or reject the reimbursement.
APE can approve or reject the reimbursement.
CFO can approve or reject the reimbursement.
REQUEST BODY (JSON)
userId , status .
APPROVAL RULE
In the EMP's view, the status turns APPROVED only once their RM and one of the APEs have
approved it.

Code checking guidelines
Since AI tools are allowed, we pay close attention to how you built the project — not just the final
result. A few things we will be looking at:
1. We will analyze your code the way it was built, so write it the way you would actually ship it.
2. We expect a clean commit history that lets us follow how you went from setting up the project to
implementing each feature.
3. A high number of commits is not what we are after — clean, structured, incremental
project building is.
4. Our tester will run your backend against an API test suite created specifically for this assignment.
5. You will be shortlisted for the next round only if we can see that you used your own problem-
solving to frame the solution and leveraged AI to write the code.
6. Maintain an incremental prompt-logs.txt file in the project root. Append every prompt you
use to it as you go — it should grow incrementally and be committed along with each commit.
How you'll be evaluated
There is no single "correct" submission. We are reading your code to understand how you build
software. In particular:
Code quality and structure — clear naming, sensible file organization, and code another
engineer could pick up and extend.
Architecture and design — the right separation of concerns and faithful adherence to SOLID
principles.
Correctness — the endpoints behave exactly as specified, with the access rules enforced per
role.
Edge-case coverage — the more thoughtfully you handle the awkward and unhappy paths, the
higher you score. This is where strong candidates separate themselves.
All the best — we are looking forward to reading your work.
